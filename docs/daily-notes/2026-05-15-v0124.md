# 2026-05-15 — Dashboard avec sous-onglets Mesures (défaut) / Bilan complet (v0.1.24)

## Décision UX

Le Dashboard d'un client devient à 2 niveaux :
1. **Top tabs** (`ClientDetailLayout`) — Dashboard / Bilans / Mesures / Notes / Historique — **inchangé**.
2. **Sub-tabs** (nouveau, dans le Dashboard) — **Mesures** (défaut) / **Bilan complet**.

Pourquoi mettre Mesures par défaut : Marie-Eve fait des prises de mesures (circonférences + plis) **plus fréquemment** que des bilans complets (qui prennent ~1h). Quand elle ouvre le dossier d'un client en suivi de remise en forme, elle veut voir « où il en est » dans son évolution corporelle — pas le dernier bilan vieux de 6 mois.

Le sous-onglet **Bilan complet** garde l'ancien Dashboard (sélecteur de bilan + donut + composites + StatsXL + radar + zones FC + forces/à travailler).

## Architecture

### Routing — `src/App.tsx`
```
/clients/:id/dashboard            → DashboardLayout (avec nav sub-tabs)
  index                           → MesuresOverview (défaut)
  /dashboard/mesures              → MesuresOverview
  /dashboard/bilan                → BilanOverview
```

L'URL reflète toujours l'onglet actif → bookmarkable, refresh-safe, retour navigateur fonctionne.

### `DashboardLayout.tsx` (nouveau)
- Nav sticky en haut avec 2 pills (`Ruler` Mesures + `ClipboardList` Bilan complet).
- Compteurs `(N)` à droite du label — somme circ + plis pour Mesures, nb de bilans pour Bilan.
- Pills cliquables même quand `count === 0` (texte grisé), pour atteindre l'empty state.
- Pill actif : fond marine + texte cream + ombre. Inactif : marine/65 + hover cream.
- **Persistance par client** dans `localStorage` clé `dashboard.subtab.<clientId>` :
  - Lecture au mount → si l'utilisatrice arrive sur `/dashboard` pile (sans suffixe), redirection vers le dernier sous-onglet visité, ou `mesures` par défaut.
  - Écriture sur chaque navigation détectée via `useLocation`.

### `MesuresOverview.tsx` (nouveau)
Contenu structuré comme demandé dans le brief :
- **Header** : « Dernière prise : 15 mai 2026 » + bouton gold `+ Nouvelle prise` (navigate vers `/mesures`).
- **2 cards principales (`BigStatCard`)** :
  - Tour de taille — gros chiffre `text-4xl`, delta colorée vs précédente, `<WaistRiskBar>` 3 segments OMS sous la valeur (si sexe renseigné).
  - % Gras corporel — Siri en gros + Brozek + mention « Durnin-Womersley » + delta.
- **4 mini-cards secondaires (`MiniStatCard`)** : Tour de hanche, Biceps moy. (G/D), Cuisse moy. (G/D), Poids — chacun avec delta.
- **Évolution dans le temps** : LineChart Recharts avec **toggle 4 métriques** (Tour taille / Tour hanche / Poids / % Gras). Sur les 12 dernières mesures (max).
- **Timeline pills** : sélecteur de session circonférences. Cliquer change `activeCirc` → recalcule les 2 cards principales + les 4 mini cards (sauf % gras qui reste sur le plus récent plis, car la timeline porte sur les circ).
- **Empty state** : si 0 mesure → gros placeholder + CTA gold `Saisir une mesure` → navigate vers `/mesures`.

### `BilanOverview.tsx` (nouveau, re-export)
Fichier-pont : `export { DashboardTab as BilanOverview } from '../tabs/DashboardTab'`. Évite de bouger `DashboardTab.tsx` (gros fichier, beaucoup d'imports internes). Si la dette de duplication des chemins gêne plus tard, on déplacera physiquement le fichier dans `dashboard/`.

## Conventions

- **Sub-tab actif sélectionné par préférence localStorage**, pas par valeur par défaut hardcodée. Marie-Eve fait souvent du suivi mesures → si elle vient de regarder un client en mesures, le prochain client s'ouvre aussi en mesures. Si elle bosse sur une session bilan, ça reste en bilan. Comportement attendu d'apps modernes (Notion, Linear…).
- **Compteurs cliquables même si 0** : empty state utile pour onboarder un client neuf (« je peux créer un bilan ici » — vs « pourquoi ce pill est désactivé ? »).
- **Compteur Mesures = circ + plis** : du point de vue de Marie-Eve, ce sont 2 types de sessions de mesure pas vraiment distinguées dans le langage courant. Si elle préfère 2 compteurs distincts (« 5 circ · 3 plis »), on adaptera.

## Dette / à garder en tête

- **Indicateur du compteur Mesures** : circ + plis additionnés. Si Marie-Eve trouve ça flou, séparer en `Mesures (7c · 7p)`.
- **`BilanOverview` est un re-export, pas un déplacement** : le code source de l'overview Bilan vit toujours dans `src/pages/client/tabs/DashboardTab.tsx`. Confus pour quelqu'un qui découvre. À déplacer physiquement quand on aura besoin de toucher ce fichier substantiellement.
- **Timeline circ ↔ % gras désynchronisés** : cliquer une session circ change les cards Tour de taille / hanche / etc., mais la card % Gras reste sur la dernière prise de plis (qui est une saisie séparée). Pas grave en pratique (les 2 types de mesures ne sont pas faits le même jour), mais à surveiller — éventuellement faire un « pair circ + plis à la même date » et le sélectionner d'un coup.
- **Empty state distingué par sous-onglet** : MesuresOverview a son propre empty state. BilanOverview a déjà le sien (inchangé). Si les deux sont vides en même temps (client tout neuf), Marie-Eve verra le placeholder Mesures par défaut — invitant à saisir une mesure plutôt qu'à importer un bilan. C'est volontaire (fréquence de saisie).
- **Pas d'indication visuelle « tu reviens sur ce client en sous-onglet bilan »** : si Marie-Eve atterrit sur Bilan complet à cause du localStorage, elle ne sait pas immédiatement pourquoi. Le pill actif est visible, donc ça reste lisible — mais si quelqu'un trouve ça déroutant, on pourrait flasher un badge « mémorisé » 2 secondes.
- **Tabler icons mentionnés dans le brief** : on utilise Lucide (déjà importé partout). `Ruler` ≈ `ti-ruler`, `ClipboardList` ≈ `ti-clipboard-list`. Cohérent visuellement.
- **Test du `useEffect` redirect** : sur `/dashboard` pile-poil, la redirection vers `mesures` ou `bilan` se fait via `navigate(... replace: true)` dans un useEffect. Risque de flicker très bref (MesuresOverview rendu une frame, puis redirect vers Bilan si localStorage = bilan). Acceptable.

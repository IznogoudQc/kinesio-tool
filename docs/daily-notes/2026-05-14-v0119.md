# 2026-05-14 — Auto-calculs complets + UX polish (v0.1.19)

## Travail réalisé

### Module central — `src/lib/bilan-computed.ts`
Une fonction `computeBilan(raw, profile)` qui retourne **tous** les calculs dérivés (anthropo + aérobie + musculo + 5 composites + score global). Délègue aux helpers existants (`body-fat-calculator`, `vo2max-calculator`, `norms/calc`, tables ACSM/CPAFLA) — pas de duplication. `mergeComputedIntoBilan(raw, computed)` injecte les valeurs au save.

Nouveaux calculs ajoutés cette release :
- `poidsOptimalMaxKg` = 25 × (taille/100)² (limite IMC normal).
- `ratioTailleHanche` = tour_taille / tour_hanche.
- `fcZones` = { z60, z65, z70, z75, z80, z85, z90 } (% de FC max prédite).

### Échelle CSEP — `CATEGORY_TO_SCORE` retoolé
- A_AMELIORER → 0.5 (était 1)
- ACCEPTABLE → 1.5 (était 2)
- BIEN → 2.5 (était 3)
- TRES_BIEN → 3.5 (était 4)
- EXCELLENT → 4.5 (était 5)
- Bornes `scoreToCategory` re-calculées : 1.0 / 2.0 / 3.0 / 4.0.

C'est cohérent avec une jauge 5-points (un score moyen tombe au milieu d'une tranche). Tests `scoring.test.ts` mis à jour.

### Sticky synthesis cards (`BilanSynthesisCards`)
- Wrapper sticky `top-0 z-10 bg-cream/95 backdrop-blur-sm py-3 -mx-6 px-6` ajouté dans `BilanForm` quand `showSynthesis` + variant `light`.
- Chaque card : gros chiffre 3xl centré, **jauge ●●●○○** (5 points remplis selon score arrondi), pill catégorie via `<CategoryBadge>`, **indicateur ▲ vert / ▼ rouge / = stable** vs bilan précédent (delta ≥ 0.05).
- Nouvelle prop `previous?: BilanComputed` pour passer la synthèse du bilan d'avant.
- Placeholder « Saisissez taille + poids + VO2max pour voir la synthèse se calculer » quand tous les scores sont null.

- « INDICES (IMPORTÉS) » → « **INDICES (CALCULÉS)** » dans `bilanFields.ts`.
- Les 4 indices (`score_composition`, `indice_sante_dos`, `score_musculo_global`, `score_global`) passés en `type: 'computed'` → toujours read-only. Hint « Cf. cards de synthèse — composition / aérobie / dos / musculo ».

La synthèse vit en haut, c'est elle qui sert de référence visuelle. Si Marie-Eve veut qu'on persiste aussi ces indices dans `bilan.data`, ce sera une bascule dans `mergeComputedIntoBilan`.

### Pré-remplissage depuis bilan précédent (`CreateBilanModal`)
- `useEffect` : récupère le bilan le plus récent du client au montage.
- Bannière gold cliquable sous le titre : « ↑ Pré-remplir avec les valeurs du bilan du DATE — N valeurs disponibles à reprendre ».
- Clic → `<PrefillModal>` : liste 18 champs candidats avec checkboxes et aperçu de la valeur.
- **Pré-cochés** (structurels, changent peu) : taille, 6 plis cutanés, tour taille/hanche, FC repos, PA sys/dia.
- **Non pré-cochés** (performance, change à chaque bilan) : poids, VO2max, push-ups, sit-ups, saut, flexion, endurance.
- Bouton « Pré-remplir (N) » → patche `data` côté parent et ferme le modal.

## Dette / à garder en tête

- **`allowImportingTsExtensions: true` ajouté à `tsconfig.web.json`** : nécessaire pour que `bilan-computed.test.ts` puisse importer `bilan-computed.ts` (qui lui-même importe ses dépendances avec extensions `.ts` pour rester compatible avec `node --test`). Avec `composite: true`, il fallait aussi ajouter `noEmit: true`. Pas d'impact sur Vite (qui résout indifféremment avec ou sans extension). Le tsconfig.node.json reste inchangé.
- **Indices CALCULÉS pas persistés** : les 4 champs `score_*` / `indice_*` restent vides en saisie manuelle. Pour Marie-Eve, la synthèse en haut est la source de vérité visuelle. Si elle remonte la confusion, ajouter au `mergeComputedIntoBilan` :
  ```ts
  next.score_composition = computed.composition.score
  next.indice_sante_dos = computed.backHealth.score
  next.score_musculo_global = computed.musculoGlobal.score
  next.score_global = computed.overall.score
  ```
- **`deriveBilanFields` est maintenant un wrapper trivial** autour de `computeBilan + mergeComputedIntoBilan`. Il reste exporté pour rétrocompatibilité (CreateBilanModal, BilanDetailTab). Pas urgent de retirer.
- **Comparaison ▲▼** : se base sur le score 0-5 (toujours « plus haut = mieux ») — pas besoin d'inverser pour les indices « lower is better » comme le suggérait le brief, parce que la conversion catégorie→score normalise déjà ça (un IMC à 32 catégorisé A_AMELIORER donne 0.5, donc plus c'est haut, mieux c'est).
- **Test E2E manuel** à faire par Marie-Eve : créer un client Nicholas (H 48), saisir 176/99.8/Bruce 13:33/plis 10-7-25-33/saut 48 → vérifier en temps réel que tous les chiffres matchent (32.2, 49, 14, 174, 5380, 30.2, score global ~2). Vérifier la sticky synthèse pendant le scroll. Re-ouvrir un 2e bilan → vérifier que la bannière de pré-remplissage apparaît, cliquer, valider la pré-coche par défaut.
- **Animation** : on a mis `transition-all duration-300` sur les cards mais pas sur les chiffres eux-mêmes (les `text-3xl font-bold` valeurs). React re-render gère le visuel, pas d'animation FLIP. À voir si Marie-Eve trouve l'effet trop sec.
- **Pré-remplissage modal** : pas d'option « tout cocher / tout décocher ». Si elle veut transférer 18 valeurs, c'est lourd. Pas urgent — on verra à l'usage.

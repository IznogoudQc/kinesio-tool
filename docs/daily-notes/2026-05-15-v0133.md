# 2026-05-15 — Objectif « niveau suivant » sur les hero stats (v0.1.33)

Remplace les sparklines des 4 `StatCardXL` du dashboard par un indicateur d'objectif motivant : « +1.5 ml/kg/min pour atteindre Excellent » ou un trophée quand la catégorie maximale est déjà atteinte.

## Pourquoi

Les sparklines des hero cards (VO2max / IMC / % gras / tour de taille) montraient l'historique des 8 derniers bilans, mais l'info était redondante : le `ProgressionChart` juste en dessous fait déjà ça, en plus grand et avec toggle de métrique. Marie-Eve a remonté que la sparkline n'était lue par personne — alors qu'un **objectif chiffré** sur la card répondrait directement à la question que le client pose : « combien il me reste pour passer dans la catégorie d'au-dessus ? ».

## Module — `src/lib/norms/index.ts`

Nouvelle fonction publique :

```ts
getNextCategoryTarget(test, value, age, sex, norms) → NextCategoryTarget | null
```

Logique :
- `null` si valeur/profil invalide ou test hors barème.
- Si `current === 'EXCELLENT'` → `{ nextCategory: 'EXCELLENT', targetValue: value, delta: 0, isAtTop: true }` (le UI affiche un trophée).
- Sinon : look up `nextThresholdMap[current]` qui donne `(nextCategory, percentileKey)` :
  - A_AMELIORER → ACCEPTABLE via `p10`
  - ACCEPTABLE → BIEN via `p25`
  - BIEN → TRES_BIEN via `p50`
  - TRES_BIEN → EXCELLENT via `p75`
- `targetValue = percentiles[key]` ; `delta = round(targetValue - value, 1 décimale)`.

Pour higher-is-better : delta positif (il faut monter). Pour lower-is-better : delta négatif (il faut descendre). Le signe est porté tel quel dans le retour — c'est au consommateur de formater (`+` explicite côté UI quand delta ≥ 0).

**Pourquoi la même clé pour lower-is-better** : la table `NormPercentiles` inverse les **valeurs** (p10 > p25 > … > p90) mais pas la **signification** des clés. La borne haute de la catégorie courante reste `pXX`.

## UI — `StatCardXL` refactorée

Suppressions :
- Imports `Line, LineChart, ResponsiveContainer` de recharts.
- Props `history?: (number | null)[]` et `sparklineColor?: string`.
- Bloc de rendu `{sparkData.length >= 2 && …}` en bas de la card.

Ajouts :
- Imports `Trophy` (lucide), `getNextCategoryTarget` et `CATEGORY_LABELS` (norms).
- Calcul `nextTarget = test && hasValue && age && sex ? getNextCategoryTarget(...) : null`.
- Rendu sous `<CategoryRangeBar>` (séparateur `border-t border-cream-dark/30`) :
  - **`isAtTop: true`** → trophée gold + label « Niveau maximal atteint ».
  - **sinon** → en-tête `OBJECTIF NIVEAU SUIVANT`, ligne `+X.X unit pour atteindre <Catégorie>`. Le `+` est explicite pour les delta positifs.

Aucun changement sur le wrapping `MetricSelectable` (mode IA inchangé).

## Dette / à garder en tête

- **Pas de format adaptatif `delta=0`** : si la valeur du client est **exactement** sur le seuil (ex. push-ups = 17 = p25 → catégorie BIEN, next = TRES_BIEN, target = p50), le delta n'est jamais 0 sauf si `isAtTop`. Mais théoriquement si on était `current=BIEN` et value=p50 (la frontière), on est déjà TRES_BIEN — donc impossible d'avoir delta=0 en pratique hors EXCELLENT. Pas un bug, mais à valider si Marie-Eve trouve une frontière flottante.
- **Pas de UI mobile / responsive distinct** : la card est déjà compacte, le bloc objectif s'empile bien. Pas de breakpoint spécial à ajouter.
- **Le « delta » n'est pas affiché en pourcentage** : on affiche la valeur brute (`+1.5 ml/kg/min`) plutôt qu'un pourcentage. Choix volontaire — un objectif chiffré dans l'unité est plus parlant à un client « 1.5 ml/kg/min de plus » que « 3 % de mieux ». Si Marie-Eve veut les deux, ajouter un sous-texte.
- **Le seuil affiché ne change pas en cours d'année** : on cible la borne basse de la catégorie supérieure. Pas de notion de « doubler la marge de sécurité » pour éviter de retomber. Si Marie-Eve veut un objectif plus ambitieux, on pourrait viser p75 + 10 % ou similaire — sur demande.
- **Pas de comparaison entre bilans** : on ne montre pas « tu étais à 25, tu es à 30, plus que +5 pour Très bien ». La trajectoire reste dans le `ProgressionChart`. À fusionner si Marie-Eve trouve les deux infos disjointes.
- **`sparkData` retiré** : le code qui filtrait `null/undefined` sur l'historique disparaît. Si on veut un jour réintroduire un mini-graph (par ex. une barre de progression « tu as bougé de X depuis ton 1er bilan »), il faudra reconstruire la logique côté card.

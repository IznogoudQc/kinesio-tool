# 2026-05-15 — Onglet Mesures (Dashboard) enrichi (v0.1.27)

Refonte du contenu de `MesuresOverview` avec 6 améliorations demandées par le brief.

## 1. Fix bug « Première mesure » sur les cards secondaires

### Cause
L'ancien code lisait `previousCirc?.bicepsG` directement — si la mesure juste avant l'active avait ce champ à `null`, la « previous » remontait `null` → MeasureDelta affichait « Première mesure » même quand 5 mesures plus loin une valeur existait.

Variante du bug pour les moyennes G/D : si la mesure précédente n'a que `bicepsG` mais pas `bicepsD`, `avgGD()` renvoyait la moyenne sur 1 valeur — pas idéal pour la comparaison.

### Fix
Nouveau helper `findPreviousValue<T>(rows, activeIdx, getter)` qui remonte la liste **triée par date desc** jusqu'à trouver une ligne où le getter renvoie un nombre fini. Permet de :

- Skipper les mesures intermédiaires où le champ est null.
- Conserver la sémantique « mesure chronologiquement précédente *qui a une valeur pour ce champ* ».
- Marcher de la même façon pour les valeurs simples (`r.taille`), les agrégats (`avgGD(bicepsG, bicepsD)`) et les valeurs dérivées (ratio T/H).

```ts
function findPreviousValue<T>(rows, activeIdx, getter) {
  for (let i = activeIdx + 1; i < rows.length; i++) {
    const v = getter(rows[i])
    if (typeof v === 'number') return { value: v, date: rows[i].date }
  }
  return null
}
```

Les 4 cards secondaires + la nouvelle card Ratio T/H utilisent ce helper. Plus de « Première mesure » erroné.

## Dette / à garder en tête

- **`findPreviousValue` ne va PAS plus loin que la liste reçue** : si circList a 7 entrées et qu'on cherche le bicepsD de l'index 6 (la plus ancienne), pas de précédent → null → « Première mesure » légitime. Pas un bug, mais à comprendre.
- **`trend90` sur la métrique chartMetric active** : si Marie-Eve toggle vers « % gras » et que tous les plis sont > 90j, le label de tendance disparait. Voulu (la fenêtre 90j n'a pas de données pertinentes).
- **Ratio T/H seuils OMS** : déjà encodés dans `who.ts` (v0.1.22). Pas de changement.
- **Mini-graph % gras dans la section Plis** : pas de filtre de période — utilise tous les points de `plisList`. Si on veut un filtre cohérent, c'est ajoutable mais peu utile à l'usage (les plis sont saisis ~tous les 3-4 mois).
- **`AllMeasuresDetails` rend toujours les 13 circonférences même si plusieurs sont vides** : volontaire pour transparence (« on a saisi ou pas ? »). Si Marie-Eve veut filtrer les vides, easy fix : `.filter(f => active[f.key] !== null)`.
- **Performance** : `findPreviousValue` boucle pour chaque card (12 dans le panneau détaillé + 5 secondaires + ratio). C'est O(N) par champ — pour N=7-20 mesures, pas un problème. Pour 200+ on optimiserait avec un précalcul indexé.
- **Layout responsive** : sur petit écran, la grille `lg:grid-cols-5` retombe à 2 cols, donc la card Ratio T/H se retrouve seule sur sa ligne. À voir si visuellement c'est OK, sinon passer à `lg:grid-cols-3` + 2 rangées.
- **Le brief mentionnait « par ordre d'insertion vs date »** : le bug n'était pas là (l'IPC trie déjà `desc(date), desc(createdAt)`), mais la formulation suggérait qu'on devrait vérifier. Confirmé — l'ordre est bien chronologique.

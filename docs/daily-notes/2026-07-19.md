## v0.8.4 — Import .doc : hanche lue à 5 au lieu de 112

**Symptôme** (Marie / Nicholas) : à l'import du bilan `.doc` du logiciel d'origine, la **Hanche** apparaissait
à **5** dans Mesures (via la synchro bilan→mesures) au lieu de **112,0 cm**.

**Diagnostic** : reproduit la conversion réelle `.doc → .docx` (LibreOffice) puis l'extraction **mammoth**
(exactement ce que voit `bilan-parser.ts`). Le texte contient deux fois « Hanche » :
- en-tête anthropométrique « **Ratio Taille/Hanche** », suivi de la taille « **5' 9"** 175,0 cm » ;
- section « **Circonférences** » plus bas : « Hanche » → « **112,0** ».

Le regex `/Hanche\s*[\n\r]+\s*(\d+…)/` accrochait le **premier** match (l'en-tête) et captait le « 5 » de « 5' 9" ».

**Correctif** : lookbehind `(?<![/\wÀ-ÿ])Hanche…` → ignore « …/Hanche » et vise le libellé isolé de la
section Circonférences. Appliqué aux **deux** extractions (courante l.180 + historique l.258).

**Vérif** : test de non-régression `electron/lib/bilan-parser.test.ts` (fixture inline reproduisant le piège) +
intégration sur le vrai `.docx` → `tour_hanche = 112`, taille 175 et tour de taille 99 inchangés.
Suite complète 295 pass, tsc node+web propres, build OK.

**Nettoyage** : suppression de `test-files/test-parser.mjs` (script de debug jetable : lisait un fichier
disparu → `node --test` échouait, et embarquait une copie périmée/buggée du parser). `tsconfig.node.json`
exclut désormais `electron/**/*.test.ts` du build.

## v0.8.5 — % de gras : « Très bien » vs « En santé » → une seule description

**Symptôme** (Marie / Nicholas) : dans le rapport, le même % de gras (23,1 % homme) était décrit
« **Très bien** » sur la carte composite et « **En santé** » dans la grille détaillée.

**Cause** : deux systèmes de classification pour le même champ — percentile ACSM (À améliorer…Excellent)
vs grille de risque de Marie (Risques potentiels / Optimal / En santé / …). Le document HTML interactif
excluait déjà `bodyFat` du percentile (grille seule) ; le PDF, lui, affichait encore le percentile dans
`CompositeBreakdown` → d'où la divergence PDF vs HTML relevée au départ.

**Décision (utilisateur)** : la **grille de Marie** devient la référence unique du % de gras, **libellé +
score**.

**Implémentation** :
- `bodyFatGridRating(pct, sex)` dans `body-fat-risk.ts` → `{ label, category }` (zone→cote :
  optimal=EXCELLENT, sante=TRES_BIEN, potentiel=BIEN, modere=ACCEPTABLE, eleve=A_AMELIORER).
- `bilan-computed.catFor` : `pourcentage_gras` coté via la grille (impacte composition + composite bodyFat).
- `ReportPage.CompositeBreakdown` : libellé de zone pour le % de gras.
- ADR 0024.

**Vérif** : suite complète **298 pass** (+3), test composition Nicholas inchangé (30,2 % → « Risques
modérés » → ACCEPTABLE = ancien percentile), tsc web+node propres, build OK.

**Hors scope (signalé)** : les cartes composites 0-4 du dashboard (`BilanSynthesisCards`, « % gras
corporel ») gardent le vocabulaire 0-4. Zones extrêmes (trop maigre) : cotation indicative à valider.

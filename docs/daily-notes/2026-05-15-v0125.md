# 2026-05-15 — Fix routing nested context dans Dashboard tabs (v0.1.25)

## Bug
La v0.1.24 a introduit un wrapper `<DashboardLayout>` qui rend `<Outlet />` pour ses sous-routes (`mesures` / `bilan`). Mais `<Outlet />` **sans `context={...}`** ne propage **pas** le `useOutletContext()` du parent — il en démarre un nouveau, vide.

Résultat : dans `MesuresOverview` et `BilanOverview` (qui héritent du contexte de `<ClientDetailLayout>`), `useClient()` retournait `undefined` et l'app crashait au premier accès à `client.<champ>`.

## Fix

### `src/pages/client/ClientDetailLayout.tsx`
- Exporté l'interface du contexte : `export interface ClientOutletContext { client: Client }`.
- Ajouté un helper public : `useClientOutletContext()` qui retourne le contexte tel quel — pratique pour les couches intermédiaires qui veulent juste le re-forwarder sans le décomposer.

### `src/pages/client/dashboard/DashboardLayout.tsx`
- Récupère le contexte parent : `const parentContext = useClientOutletContext()`.
- Le passe à l'Outlet : `<Outlet context={parentContext} />`.

C'est exactement le pattern documenté par React Router : « Si vous avez un layout intermédiaire entre un parent qui fournit un context et des enfants qui le consomment, le layout doit explicitement le re-forwarder. » `<Outlet />` n'a pas de magie d'héritage de contexte — le context d'Outlet est local à cet Outlet.

## Dette / à garder en tête

- **Pattern de re-forward** : si on ajoute d'autres couches imbriquées (par ex. un `<MesuresLayout>` qui aurait ses propres sous-onglets — ce n'est pas prévu mais imaginable), il faudra refaire ce re-forward à chaque niveau. Pas de mécanisme automatique côté React Router.
- **`useClient()` vs `useClientOutletContext()`** : les deux sont exportés depuis `ClientDetailLayout.tsx`. Le premier décompose pour ne donner que `client`, le second donne le contexte complet. Si on enrichit `ClientOutletContext` plus tard (par ex. `onClientUpdated`), les composants intermédiaires utilisant `useClientOutletContext()` se mettront à jour automatiquement ; ceux qui n'avaient besoin que de `client` resteront sur `useClient()`.
- **`useClient()` typage strict** : retourne `Client` (pas `Client | undefined`). Si le contexte n'est pas fourni (= bug de routing comme celui-ci), l'erreur survient à l'accès, pas au typage. Idéalement on aurait un `useClient()` qui throw « ClientOutletContext non fourni — êtes-vous dans une sous-route de /clients/:id ? » — à faire si on a d'autres bugs similaires.

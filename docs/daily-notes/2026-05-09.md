## Procédures spéciales (machine sans Visual Studio C++ Build Tools)

### Premier setup (`npm install`)

Le `.npmrc` dirige `prebuild-install` vers les prebuilds Electron. Si `npm install` échoue quand même :

```bash
npm install --ignore-scripts
node node_modules/electron/install.js
npx electron-builder install-app-deps
```

### `npm run dist` — workaround winCodeSign

Sur Windows sans Developer Mode activé, `electron-builder` ne peut pas extraire le cache `winCodeSign` (symlinks macOS nécessitent `SeCreateSymbolicLinkPrivilege`).

Workaround (à exécuter une seule fois) :

```bash
python3 scripts/fix-wincodeSign-cache.py
```

Ou manuellement avec Python :

```python
import shutil, os
# Trouver un répertoire de tentative partielle dans le cache
src = r'C:\Users\njean\AppData\Local\electron-builder\Cache\winCodeSign\595473967'
dst = r'C:\Users\njean\AppData\Local\electron-builder\Cache\winCodeSign\winCodeSign-2.6.0'
if os.path.exists(dst): shutil.rmtree(dst)
shutil.copytree(src, dst)
```

**Note** : Ce workaround est permanent — une fois le répertoire `winCodeSign-2.6.0` créé dans le cache,
`npm run dist` fonctionne normalement. La solution définitive est d'activer **Windows Developer Mode**
(Paramètres → Système → Pour les développeurs → Mode développeur).

## v0.1.6 — Importation de bilans depuis .docx

### Schéma DB

- Table `bilans` (déjà présente dans `db/schema.ts`, migration `0002_lively_amazoness.sql`) :
  `id` (UUID), `client_id` (FK → `clients`, ON DELETE cascade), `date` (ISO 8601), `data` (blob JSON
  des valeurs extraites), `source` (`import_docx` | `manuel`), `created_at`. `npm run db:generate`
  confirme : schéma en phase, rien à migrer.

### Parsing .docx — `electron/lib/bilan-parser.ts`

- `mammoth` (`extractRawText`) → texte brut, puis regexes sur les patterns du logiciel CPAFLA :
  - **Date du bilan** : premier `\d{1,2} <mois fr> 20\d{2}` dans l'en-tête.
  - **Chevrons `►`** : `chevronValue('Pourcentage de Gras', …)` → `30,2`, idem VO2max, indices santé dos /
    musculo / global, PA systolique/diastolique, FC repos, push-ups, redressements, saut vertical,
    puissance jambes, flexion tronc, endurance dos.
  - **Anthropométrie** : section `Taille / Poids / IMC / Circonf.` → cm/kg/kg·m⁻², plis cutanés
    (triceps, biceps, sous-scap, iliaque, mollet, cuisse), tour de hanche.
  - **Type de test aérobie** : heading `Aptitude Aérobie <nom du test>` (ex. « Tapis Roulant de Bruce »).
  - **Historique** : ancré sur `Indice de masse corporelle` (1ʳᵉ ligne du tableau de comparaison) ; les
    dates de colonnes le précèdent immédiatement (cluster serré). Une ligne par mesure (IMC, composition,
    % gras, tour de taille, plis, hanche) → un `ExtractedBilan` par date antérieure.
  - Nombres au format français (virgule décimale) via `parseFrenchNumber`.
- Retourne `{ current: { date, data }, history: ExtractedBilan[] }`.

### Service + IPC (API-ready)

- `electron/ipc/bilans.ts` — handlers : `bilans:pick-docx` (dialog natif `showOpenDialog`, filtre `.docx`,

  `bilans:create`, `bilans:update`, `bilans:list` (récents en premier), `bilans:get-by-id`. Validation

- Preload : `window.api.bilans.{ pickDocxFile, parseDocx, create, update, list, getById }`.

## v0.1.7 — Correctifs import : déduplication, suppression, nettoyage des doublons

**Problème** : l'import d'un `.docx` créait des doublons `(client_id, date)` parce que les bilans
historiques se chevauchent d'un fichier à l'autre. Trois correctifs.

### 1. Déduplication à l'import

- Nouveau handler IPC `bilans:import` (`{ clientId, bilans: [{ date, data }] }`) qui, pour chaque bilan
  extrait : INSERT si pas d'existant sur `(client_id, date)` ; sinon UPDATE si le nouveau a *plus* de
  champs renseignés que l'existant (comptage des valeurs non vides du JSON `data`) ; sinon ignoré.
  Retourne `{ imported, updated, skipped }`.
- `src/services/bilans.ts` : `importBilans(clientId, items)`. `ImportBilanModal` n'appelle plus `create`
  en boucle → un seul `importBilans`, puis affiche un récap (« X importés, Y mis à jour, Z ignorés (déjà
  présents) ») avec un bouton « Fermer ». Toast équivalent dans `BilansTab`.
- `bilans:create` (unitaire) conservé pour la future saisie manuelle.

### 2. Suppression de bilans

- Handler IPC `bilans:delete(id)` + `bilansService.delete(id)`.
- `BilansTab` : icône poubelle rouge (`Trash2`) sur chaque ligne, à droite de « Voir » → dialog de
  confirmation « Supprimer le bilan du [date] ? Action irréversible. » → suppression + rafraîchissement + toast.

### 3. Nettoyage en masse des doublons (bonus)

- Handler IPC `bilans:dedupe(clientId)` : pour chaque date ayant >1 bilan, garde le plus complet (et y

- `BilansTab` : bouton « Nettoyer les doublons » (visible seulement s'il y a des doublons, à côté

  « à garder » (le plus complet) vs « à supprimer », bouton « Tout nettoyer » → `dedupe` + toast.

## v0.1.7 — Import des bilans au format `.doc` (conversion automatique → `.docx`)

**Besoin** : certains anciens bilans de Marie-Eve sont au format `.doc` (binaire), que `mammoth` ne sait
pas lire. On les convertit en `.docx` à la volée avant l'analyse.

### Module de conversion — `electron/lib/doc-converter.ts`

- `convertViaWord(docPath, outputPath)` : pilote Microsoft Word en COM via un script PowerShell
  (`New-Object -ComObject Word.Application` → `Documents.Open` en lecture seule → `SaveAs(..., 16)` → `Quit`),
  exécuté avec `spawn('powershell.exe', ['-NoProfile', '-NonInteractive', '-Command', …], { windowsHide: true })`.
  Windows uniquement.
- `convertViaLibreOffice(docPath, outputDir)` : `spawn(soffice, ['--headless', '--convert-to', 'docx',
  '--outdir', outputDir, docPath])`. `soffice` cherché dans les emplacements habituels (`C:\Program Files\
  LibreOffice\program\soffice.exe`, `…(x86)…`, `/usr/bin/soffice`, `/usr/local/bin/soffice`,
  `/Applications/LibreOffice.app/Contents/MacOS/soffice`).
- `convertDocToDocx(docPath)` (dispatcher) : essaie Word d'abord (rapide, sur Windows) ; en cas d'échec,
  repli automatique sur LibreOffice ; si aucun n'est dispo → erreur claire (« ni Microsoft Word ni
  LibreOffice ne sont installés… LibreOffice est gratuit : https://www.libreoffice.org … »). Écrit un
  fichier temporaire `kinesio-<uuid>.docx` dans `os.tmpdir()` ; logs via `electron-log`.

### Intégration — `electron/ipc/bilans.ts`

- `bilans:pick-docx` : filtre du dialog natif passé à `extensions: ['doc', 'docx']` ; ne lit plus les octets
  → renvoie maintenant `{ canceled, fileName, filePath }` (chemin absolu).
- `bilans:parse-docx(clientId, filePath)` : vérifie l'existence et l'extension (`.doc` | `.docx`) ; si `.doc`
  → `convertDocToDocx` → parse le `.docx` temporaire → `unlinkSync` du temporaire dans le `finally`.
- Conséquence : suppression du transfert des octets sur le bus IPC (plus léger, plus sûr). `PickedDocx`
  (`env.d.ts`) : `data: Uint8Array` → `filePath?: string`. `bilansService.import_docx(clientId, filePath)`.

### UI

- `BilansTab` : libellé du bouton → **« Importer un bilan (.doc, .docx) »** ; empty state → « bilan .doc ou
  .docx » ; idem CTA de l'état vide du dashboard. En cas d'échec de conversion, le toast affiche le message
  d'erreur complet (lien LibreOffice inclus), nettoyé du préfixe « Error invoking remote method '…': » via
  `cleanIpcError`.

### Test manuel

- `.doc` + Word installé → conversion via Word (rapide). `.doc` + seulement LibreOffice (simuler en
  renommant temporairement `winword.exe`, ou sur une machine sans Office) → repli LibreOffice. Ni l'un ni
  l'autre → message d'erreur clair avec le lien d'installation. `.docx` → comportement inchangé (pas de
  conversion).

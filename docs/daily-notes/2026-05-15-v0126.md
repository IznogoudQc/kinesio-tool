# 2026-05-15 — Fix Rules of Hooks dans MesuresOverview (v0.1.26)

## Bug

`MesuresOverview` violait la règle des hooks de React :

```
hooks (useState ×6, useEffect, useMemo ×2)
  → if (loading) return … ← early return
  → if (error) return …
  → if (!hasAnyMeasure) return …
  → useMemo(chartData) ← hook APRÈS early return → React warning
```

Au premier render (avant le chargement asynchrone), `loading === true` donc le `useMemo(chartData)` n'était pas appelé. Au second render (après `setLoading(false)`), il l'est. React détecte « Rendered more hooks than previous render » et plante.

## Fix

Tous les hooks ont été déplacés **avant** les early returns, dans l'ordre :

```
hooks (useState ×6, useEffect, useMemo ×2 anciens, useMemo METRICS [nouveau], useMemo chartData [déplacé])
  ↓
  if (!client) return    ← defensive (au cas où context tarde)
  if (loading) return
  if (error) return
  if (!hasAnyMeasure) return
  ↓
  return <Content />
```

Le `useMemo(chartData)` (anciennement ligne 256, après les returns) est maintenant déclaré en haut. Pour respecter `react-hooks/exhaustive-deps` correctement, j'ai aussi extrait `METRICS` dans un `useMemo` plutôt que de le redéclarer à chaque render — il dépend de `lenLabel` et `wLabel` qui dérivent de `client.unitLength` / `unitWeight`.

## Robustesse — `client` traité comme potentiellement undefined

J'ai ajouté un cast défensif `useClient() as Client | undefined` et un garde `if (!client) return <chargement…>` avant de toucher aux propriétés. Au cas où le contexte Router prend une frame à se propager.

L'`useEffect` qui charge les mesures dépend maintenant de `clientId = client?.id` au lieu de `client.id` — pas de crash si le contexte tarde.

## Pattern à respecter pour les futurs composants

```tsx
export function MyTab() {
  // 1. TOUS les hooks en haut, inconditionnels.
  const ctx = useSomeContext()        // peut être undefined
  const [x, setX] = useState(...)
  useEffect(() => {
    if (!ctx) return                  // garde DANS le hook, pas avant
    // ...
  }, [ctx])
  const memo = useMemo(() => {
    if (!ctx) return null             // garde DANS le hook
    return compute(ctx)
  }, [ctx])

  // 2. Early returns APRÈS le dernier hook.
  if (!ctx) return <Loading />
  if (someError) return <Error />

  // 3. Render réel.
  return <Content />
}
```

## Dette / à garder en tête

- **`useClient()` retourne toujours `Client` (pas `Client | undefined`)** dans le typage. Le cast `as Client | undefined` dans MesuresOverview est une défense contre un context manquant qui *en théorie* ne devrait pas arriver après le fix v0.1.25. Si on veut être strict, on peut soit :
  - Durcir `useClient()` pour throw « ClientOutletContext manquant — êtes-vous dans /clients/:id/* ? » (refus rapide, message clair).
  - Changer son retour en `Client | undefined` (oblige tous les consommateurs à gérer le cas — refactor large).
  - Garder l'état actuel et faire confiance au contrat de routing (== aujourd'hui, sauf le cast défensif local).
- **Pattern hooks-then-returns** : à rappeler dans les revues de code, et `react-hooks/rules-of-hooks` détecte normalement ce genre d'erreur — mais le linter ne tournait pas sur ce fichier au moment du dev. Si on veut s'assurer, on peut ajouter `eslint-plugin-react-hooks` à la config locale (déjà mentionné dans les 3 erreurs préexistantes de `out/renderer` — c'est ce plugin qui manque dans la config root).
- **Pas de test unitaire pour ce genre de bug** : les Rules of Hooks ne se voient qu'au runtime (warning console + plantage du composant). Un test E2E type Playwright qui ouvre la page et vérifie l'absence d'erreur console serait la vraie ceinture de sécurité. À l'usage si on en accumule.

# 2026-05-14 — Percentiles ACSM + comparaison vs population (v0.1.20)

## Travail réalisé

### Refactor type — `NormRange` porte des percentiles
`thresholds: {aAmeliorer/acceptable/bien/tresBien}` remplacés par `percentiles: {p10, p25, p50, p75, p90}` (cf. ADR [[0006-percentiles-vs-categories]]). Catégorie dérivée des cutoffs P10/P25/P50/P75. P90 sert au calcul fin du percentile (≥80) et à la zone EXCELLENT.

### Migration mécanique des tables ACSM (`acsm.ts`)
- `aAmeliorer → p10`, `acceptable → p25`, `bien → p50`, `tresBien → p75`
- `p90 = 2·p75 - p50` (extrapolation linéaire douce — adéquate pour des distributions ~normales)
- **3 entrées recalibrées** sur les valeurs ACSM 11e éd. publiées pour matcher les attentes du brief (Nicholas Jean, 48 ans) :
  - VO2max H 40-49 : pct(23, 30, 35, 43, 50)
  - % gras M 40-49 : pct(35, 30, 25, 20, 14) lowerIsBetter
  - Push-ups M 40-49 : pct(9, 12, 16, 22, 28)
- Les **autres** brackets conservent le mapping mécanique → catégorisation **identique** à v0.1.15.

### Nouvelle API publique — `src/lib/norms/index.ts`
- `getCategorization(test, value, age, sex, norms)` (inchangée — backend percentile)
- `getPercentile(test, value, age, sex, norms)` — interpolation linéaire 0-100 entre les 5 ancres, clampée. Gère `lowerIsBetter` en inversant le sens « performance ».
- `getPopulationAverage(test, age, sex, norms)` — retourne P50.
- `getDeltaVsAverage(test, value, age, sex, norms)` — `{deltaPct, isBetter}`. Pour `lowerIsBetter`, inversion automatique pour rester intuitif (positif = mieux).

### Catégorisation re-câblée
Les `categorizeRaw()` locaux dans `scoring.ts` et `bilan-computed.ts` mis à jour pour lire `range.percentiles` au lieu de `range.thresholds`. Pas d'autre impact en aval — `computeBilan` continue de fonctionner.

## Dette / à garder en tête

- **Couleurs Tailwind `lowerIsBetter`** : pour un IMC ou un % gras élevé, le delta vs moyenne est négatif → indicateur rouge. C'est cohérent. Mais la pastille de la mini-barre suit la même couleur — donc un client en bonne santé (IMC 22) verra un point vert sous le P50. C'est OK visuellement mais à valider avec Marie-Eve si elle préfère un dégradé fixe (rouge à gauche, vert à droite).
- **CPAFLA** : architecture pluggable inchangée. `getCpaflaRange` retourne toujours `null` → l'indicateur disparaît silencieusement sur les bilans CPAFLA. Quand Marie-Eve confirmera les tables, il suffira de remplir `cpafla.ts` avec la même structure `percentiles`.
- **Tooltips** : on utilise `title=""` natif (pas de tooltip stylé). Suffisant pour Marie-Eve (usage clinique privé), mais perfectible. À envisager si le rapport PDF a besoin d'expliquer l'indicateur — probablement pas (le PDF est un livrable, pas un outil interactif).
- **Mini-barre dans synthesis cards** : déféré (choix Marie-Eve « Core »). Les cards de synthèse continuent d'afficher la jauge ●●●○○ et le delta ▲▼ vs bilan précédent — pas la position dans la population. À ajouter si demande remonte.
- **Tables non recalibrées** : pour les brackets hors M 40-49, les valeurs des percentiles sont celles auto-migrées depuis les anciens seuils. Cela peut donner des écarts de quelques points vs ACSM 11e éd. exact pour d'autres clients. Acceptable pour v0.1.20 ; recalibrage complet recommandé si Marie-Eve rapporte des incohérences sur d'autres âges.
- **Tests `norms.test.ts`** : les helpers `interpolatePercentile/getPercentile/getDeltaVsAverage` sont **dupliqués** dans le fichier de test (même pattern que `getCategorization`). Raison : `node --test` ne peut pas suivre la chaîne d'imports sans extension `.ts`. Si on modifie ces fonctions, mettre à jour les deux endroits. Au futur on pourra remplacer par un seul fichier avec `allowImportingTsExtensions` partout — déjà fait pour `tsconfig.web.json` mais pas pour `tsconfig.node.json`.
- **Calibration body fat F 40-49** : conserve les valeurs auto-migrées (pas modifié comme M 40-49). Si Marie-Eve fait un bilan pour Sabrina ou une autre femme 40-49 ans, les percentiles seront un peu décalés.

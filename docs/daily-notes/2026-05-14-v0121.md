## Update (2026-05-15 ter) — `<CategoryRangeBar>` (barre de seuils style logiciel CPAFLA)

### Composant — `src/components/CategoryRangeBar.tsx`
Visualisation inspirée du logiciel actuel de Marie-Eve : 5 segments colorés (rouge → vert foncé) côte à côte avec labels « À améliorer / Acceptable / Bien / Très bien / Excellent », seuils numériques aux jonctions (P10, P25, P50, P75), et **marqueur ▼ TOI** positionné selon le percentile de performance.

API :
```tsx
<CategoryRangeBar
  value={49}
  percentiles={{ p10: 23, p25: 30, p50: 35, p75: 43, p90: 50 }}
  unit="ml/kg/min"
  ageRange="Hommes 40-49 ans"
  variant="full"
/>
```

Props :
- `value` (`number | null`) — valeur du client.
- `percentiles` (`NormPercentiles`) — issu de `getNormPercentiles(test, age, sex, norms)`.
- `unit`, `ageRange`, `markerLabel` (défaut « TOI »).
- `lowerIsBetter` — gère le sens d'interprétation pour % gras / IMC / tour de taille / FC repos.
- `variant: 'compact' | 'full'` — `compact` = juste segments + marqueur ▲ minuscule en dessous (pour les cards denses). `full` = labels + seuils + valeur au-dessus + « TOI » en dessous.

### Logique pure — `src/lib/range-bar-position.ts`
Extraction de `calculatePosition(value, percentiles, lowerIsBetter)` dans un module sans JSX, **testable via `node --test`**. Algorithme : interpolation linéaire entre les 5 ancres (P10, P25, P50, P75, P90) avec extrapolation douce hors bornes, axe inversé pour `lowerIsBetter`. Résultat clampé à [0, 100].

### Convention d'orientation — important
J'ai retenu l'**OPTION C** discutée mentalement : la barre est toujours orientée rouge (gauche, pire performance) → vert foncé (droite, meilleure performance), pour les deux types de tests. Le marqueur représente le **percentile de fitness** :
- `lowerIsBetter=false` (VO2max, push-ups…) : marker à droite = valeur élevée = bon.
- `lowerIsBetter=true` (% gras, IMC…) : marker à droite = valeur basse = bon.

Le brief mentionnait « Si lowerIsBetter : inverser l'ordre des couleurs (excellent à gauche, à améliorer à droite) » MAIS donnait aussi « % gras 30.2 → marqueur à environ 30 % (Acceptable) ». Les deux sont incompatibles : à 30 % d'une barre rouge → vert avec Acceptable au segment 20-40 %, le rouge reste à gauche. J'ai privilégié la **cohérence sémantique** (« la barre va toujours du pire au meilleur ») sur le mapping littéral des couleurs. Le résultat : « % gras élevé = marker à gauche dans le rouge » est en fait intuitif une fois qu'on comprend que l'axe est la fitness, pas le poids brut.
